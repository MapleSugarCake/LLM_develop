Ollama Python 客户端使用说明文档

一、安装与环境配置

在使用 Ollama Python 客户端前，需完成服务端与客户端的环境准备。以下为标准配置流程：

  • Ollama 服务端安装
    您需在本地或目标服务器上安装并运行 Ollama 服务。官方支持 macOS、Linux 和 Windows 系统，可通过包管理器或一键脚本快速部署。服务默认监听 http://localhost:11434，确保该端口未被占用。

  • Python 运行环境
    客户端基于 Python 3.7+ 构建，推荐使用虚拟环境（如 venv 或 conda）隔离依赖，保障项目稳定性。

  • 依赖库安装
    客户端核心依赖为 requests 库，用于发起 HTTP 请求与处理流式响应。通过 pip 安装： ```bash pip install requests

- **网络连通性验证**  
  确保 Python 客户端能访问 Ollama 服务端 API 端点。若服务部署于远程主机，需配置正确的 `base_url` 并开放防火墙端口。

完成上述步骤后，即可初始化 `OllamaClient` 实例，开始调用模型生成接口。

## 二、基础用法示例

本节展示如何使用封装好的 `OllamaClient` 类进行基本的模型调用。该客户端基于 Python 标准库 `requests` 实现，支持与本地运行的 Ollama 服务（默认端口 `11434`）通信。

### 初始化客户端
首先创建 `OllamaClient` 实例，可指定自定义服务地址：

```python
from ollama_client import OllamaClient

# 使用默认地址 http://localhost:11434
client = OllamaClient()

# 或连接远程服务器
# client = OllamaClient(base_url="http://192.168.1.100:11434")

调用文本生成接口

使用 /api/generate 端点执行提示补全任务，适用于非对话式场景：

try:
    result = client.generate(
        model="llama3.2",
        prompt="为什么天空是蓝色的？"
    )
    print("生成结果:", result["response"])
    print("上下文长度:", len(result["context"]) if result["context"] else 0)
except Exception as e:
    print("请求失败:", e)

调用聊天对话接口

使用 /api/chat 端点实现多轮对话，通过维护消息列表保持上下文连贯性：

try:
    messages = [
        {"role": "user", "content": "你好！"}
    ]
    result = client.chat(
        model="llama3.2",
        messages=messages
    )
    print("助手回复:", result["response"])
    print("当前消息数:", len(result["messages"]))
except Exception as e:
    print("聊天请求失败:", e)

关键说明： - 所有方法均返回结构化字典，包含 response、stats 性能指标及用于会话记忆的字段。 - 建议始终使用 try-except 捕获异常，确保程序健壮性。

三、流式响应实现方式

Ollama 的 /api/generate 和 /api/chat 端点原生支持流式响应，允许客户端实时接收模型生成的每一个 token，适用于构建交互式对话界面或需要低延迟反馈的应用场景。

核心机制

  • 流式响应以 JSON 行流（line-delimited JSON） 形式返回，每行一个独立的 JSON 对象。
  • 每个数据块包含部分生成内容（response 或 message.content），最终块带有 "done": true 标志。
  • 客户端需逐行读取并解析 HTTP 响应流，直至接收到完成信号。

实现模式

通过 requests 库的 iter_lines() 方法处理流式数据，结合异常捕获确保连接安全关闭。以下为封装后的流式调用示例：

def _handle_stream_response(self, response: requests.Response) -> Iterator[Dict[str, Any]]:
    """处理流式响应，逐行解析JSON并yield每个chunk"""
    try:
        for line in response.iter_lines():
            if line:
                chunk = json.loads(line.decode('utf-8'))
                # 检查流内错误
                if chunk.get("status") == "error":
                    raise RuntimeError(f"Stream error: {chunk.get('error', 'Unknown error')}")
                yield chunk
    except json.JSONDecodeError as e:
        raise RuntimeError(f"Failed to decode JSON from stream: {e}")
    finally:
        response.close()

关键提示：
- 必须检查每个 chunk 是否包含 "done": true 以识别流结束。
- 最终响应中包含性能统计字段，如 eval_count（生成 token 数）和 eval_duration（生成耗时，纳秒），可用于计算吞吐量（token/s）。
- 即使 HTTP 状态码为 200，流内部仍可能返回错误，需持续监听 "status": "error" 字段。

四、多线程调用与并发管理

在构建高性能 Ollama 应用时，合理利用多线程机制可显著提升推理效率与系统吞吐量。需明确区分两种“多线程”场景：一是模型推理级的并行计算（控制单次请求的资源使用），二是客户端请求级的并发处理（实现多个请求的同时执行）。二者相辅相成，共同优化整体性能。

模型推理级并行：控制单次请求线程数

通过设置 options.num_thread 参数，可在运行时指定模型推理所使用的 CPU 线程数量，直接影响单个请求的生成速度与系统负载。

特性 说明
参数路径 options["num_thread"]
数据类型 整数
默认值 由系统自动决定
推荐设置 设置为物理核心数或略高，避免过度竞争

此方式适用于对延迟敏感的长文本生成任务，通过增加线程数加速 token 生成过程。示例如下：

# 使用8个线程进行模型推理
result = client.generate(
    model="llama3.2",
    prompt="请写一篇关于气候变化的科普文章。",
    options={"num_thread": 8}
)

注意：并非线程越多越好，过多线程可能导致上下文切换开销增大，反而降低性能。

客户端请求级并发：批量处理多个请求

对于需要同时处理多个用户请求的服务场景，应采用 ThreadPoolExecutor 实现客户端级别的并发调用，最大化资源利用率。

from concurrent.futures import ThreadPoolExecutor, as_completed

# 定义多个请求任务
requests_list = [
    {"type": "generate", "model": "llama3.2", "prompt": "为什么天空是蓝色的？"},
    {"type": "chat", "model": "llama3.2", "messages": [{"role": "user", "content": "你好！"}]},
    {"type": "generate", "model": "codellama:7b", "prompt": "写一个快速排序函数"}
]

# 并发执行
with ThreadPoolExecutor(max_workers=3) as executor:
    futures = [executor.submit(client.multi_request, [req]) for req in requests_list]
    for future in as_completed(futures):
        try:
            result = future.result()
            print("完成请求:", result[0].get("response", "")[:50] + "...")
        except Exception as e:
            print("请求失败:", e)

该模式特别适合批处理、压力测试或高并发 API 网关等应用场景，能有效提升单位时间内的请求处理能力。

五、会话记忆与错误处理最佳实践

在构建稳定、智能的 Ollama 应用时，有效管理对话状态与应对异常情况是核心挑战。本章将深入探讨两种会话记忆机制的设计选择，并提供一套全面的错误处理策略，确保系统在真实环境中的健壮性。

会话记忆实现方式

Ollama 提供了两条技术路径来维持多轮对话的上下文连贯性，开发者应根据应用场景选择最合适的方案。

  • /api/generate 路径：基于 context 字段
    此方法通过服务器返回的 context 数组（一组数字，代表 token 的编码）来压缩并传递对话历史。客户端需在后续请求中显式携带此字段，以延续之前的生成状态。该方式轻量高效，适用于简单的“提示-响应”循环，但对开发者要求较高，需自行管理 context 生命周期。

  • /api/chat 路径：基于 messages 数组
    此方法采用更直观的对话记录模式。客户端通过维护一个包含 role（如 user, assistant）和 content 的消息列表，完整地拼接整个对话历史。Ollama 服务端负责解析此列表并生成回复。这种方式逻辑清晰，易于调试，是实现复杂多轮对话的推荐选择。

以下表格总结了两种方式的关键差异：

特性 /api/generate 方式 /api/chat 方式
核心字段 context (整数数组) messages (对象数组)
数据形式 编码后的上下文向量 显式的角色-内容对
维护方式 需手动传递上一次响应中的 context 自动追加新的 assistant 消息至列表
适用场景 简单的文本补全、代码生成 多轮聊天机器人、复杂任务编排
可读性 低（二进制形式） 高（纯文本，便于日志与调试）

实践建议：对于新项目，强烈推荐使用 /api/chat 接口。其结构化的消息格式不仅简化了会话管理，还天然支持 system 角色设定和未来可能的 tool_calls 扩展。

错误处理最佳实践

一个健壮的客户端必须能优雅地处理各种失败场景。Ollama API 的错误可能发生在两个层面：HTTP 通信层和流式响应内部。

HTTP 层错误

这些错误由 requests 库抛出，通常表示网络或请求配置问题。

  • 404 Not Found：模型不存在。应检查模型名称拼写，并考虑先调用 /api/pull 下载模型。
  • 400 Bad Request：参数无效。需验证 JSON 结构，特别是 options 中的数值类型。
  • 连接超时/拒绝：检查 Ollama 服务是否运行，以及 base_url 是否正确。

流内错误

即使 HTTP 连接成功，流式响应中仍可能包含错误信息。

  • 在逐行解析 JSON 流时，必须检查每个 chunk 是否包含 "status": "error" 或类似字段。
  • 例如，在拉取模型时，流中可能出现 { "status": "pull failed", "error": "network timeout" }。

综合处理策略：

  1. 统一异常捕获：所有 API 调用都应包裹在 try-except 块中，捕获 requests.RequestException 及其子类。
  2. 流式监听：在 _handle_stream_response 方法中，对每一行解码后的 JSON 对象进行校验，一旦发现错误状态立即抛出异常。
  3. 幂等性设计：对于非致命错误（如网络抖动），可实现带退避策略的重试机制。
  4. 资源清理：使用 finally 块确保 response.close() 被调用，防止文件描述符泄漏。

遵循以上实践，可构建出既能保持长期对话又能从容应对故障的高质量 Ollama 客户端应用。
